マネコ家計簿を作った話

2026年7月Web / TypeScript / three.js
🐱

✍️ リード（つかみ）は自分の言葉で 「家計簿アプリって続かないよね」から始めるのが定番。ここに、あなたがこのアプリを作ろうと思った瞬間の一言を。 （例：レシートを撮るだけの家計簿を作ったけど、自分ですら開かなくなった。なぜだろう…から始まった。）

1. テーマ — なぜ「叱らない家計簿」なのか

マネコ家計簿は、3Dキャラクター「マネコ」（トゥーン調の猫）が主役の家計簿です。 普通の家計簿が「いくら使ったか」を数字で突きつけるのに対し、マネコ家計簿は 買ったものにマネコが反応し（食べる・飲む・持つ・着る）、使いすぎても叱らずにそっとアドバイスします。 ラーメンが続けば「丸くなってきた…頻度減らそ?」、飲み会が続けば「肝臓きつい〜」。 日用品や固定費には口を出しません。

✍️ ここが一番の読ませどころ：なぜこのテーマにしたか ・数字で管理する家計簿が続かなかった実体験は？
・「叱らない」にこだわった理由は？（叱られると嫌になる、という自分の感覚など）
・なぜ“猫のキャラ”だったのか。数字じゃなくキャラにした狙い。

2. サービスの説明 — 何ができるアプリか

初回ログインで、2つのモードを選びます。同じ家計データを別の見た目で使えます。

  • 🏘️ こどもモード「マネコタウン」 — 3Dの街で、おさいふ・コイン・XP/レベル・ごきげん度・きょうのチャレンジ・ちょきん箱・衣装ガチャ。家計簿をゲームのように続けられる。
  • 💼 おとなモード「マネコ家計簿」 — 「今月あと使える額」を大きく見せるヒーローカード＋予算バー＋つかいみち内訳＋最近の記録。覗きこむマネコがアクセント。

共通の下部ナビは ホーム / レポート / きろく / ちょきん / せってい。 支出は金額・カテゴリだけでなく、レシートOCR（Claudeの画像認識）で明細ごとに取り込んだり、 地図上のピンで「どこで使ったか」を残せます。家族グループでの割り勘・精算にも対応しています。

✍️ 補足するなら 実際に自分が毎日どう使っているか、お気に入りの機能を一言。

3. 設計の説明

全体構成（3層）

授業の指定に沿って、フロント＝素のTypeScript（フレームワークなしのDOM操作）、 バック＝Node.js + Express、DB＝PostgreSQL の3層で作りました。 GitHubにpushするとRenderに自動デプロイされます。

  • フロント：src/client/。402×840のスマホ画面をキャンバスに見立て、素のDOMで描画。
  • 3Dのマネコ：character.ts。モデルを読み込むのではなく、three.js で猫をプロシージャル生成（体・耳・しっぽを球や円柱から組み立て）。食べすぎた月は体型が丸くなる。
  • アドバイス：advice.ts。支出のパターン（同じカテゴリの連続など）から、叱らないメッセージを選ぶ。日用品・固定費は対象外にするルール。
  • サーバ：index.ts / api.ts / auth.ts / ocr.ts。cookieセッション認証、家族グループ、レシートOCR。

こだわった設計判断

  • 2アプリを1コードで：こども/おとなはモード切替で、同じデータを別UIで見せる設計。
  • PWA対応：manifest ＋ Service Worker で app shell をオフラインキャッシュし、ホーム画面から standalone 起動できる。?v=N でアセットをバージョニング。
  • スクロール設計：ホームは絶対配置＋scale、他タブは高さをフィットさせ内側だけスクロール、という切り分け。

✍️ ここも自分の言葉で：なぜその設計にしたか three.jsでモデルを読み込まず“組み立てた”理由（軽さ？勉強のため？）／ 2モードにした判断で悩んだこと、など。図やスクショを1枚入れると強い。

4. 自分なりにこだわった点

✍️ 箇条書きで3〜4個、あなたのこだわりを 候補：
・叱らないアドバイスの“言い回し”（キツくなりすぎない絶妙なライン）
・マネコが買ったものに反応する演出（食べる/飲む/着る）
・こどもモードのRPG感（レベル・ごきげん・ガチャ）で「続く」体験
・キャラの世界観の統一
→ それぞれ「なぜこだわったか」「どう実現したか」を一言ずつ。

5. 難しかった・悩んだこと

正直、うまくいかなかったことの方が多かったです。書ける範囲で残しておきます。

デザインのモック（HTML/CSS）を渡されたのに、最初は自己流に“解釈”して作ってしまい、 「全然違う。忠実に再現して」と差し戻された。
整ったデモアカウントでしか動作確認しておらず、新規ユーザーが最初に触る導線がバグだらけだった （DBエラーでホームごと落ちてキャラが出ない、記録までの手順が多すぎる、など）。

✍️ この失敗を“自分の言葉”で語ると記事が一気に面白くなる 上の2つはメモに残っていた実話。ここに「そのとき何を感じたか」「どう直したか」「次からどうするか」を。 先生が一番読みたいのは、まさにこの“人間が考えたこと”の部分。

6. 学びになったと感じたこと

✍️ 技術面と、それ以外の学びを分けて書くと良い 技術：three.jsでの3D、素のDOMでのSPA、PWA、OCR連携、Renderデプロイ… の中で「これは身についた」というもの。
技術以外：デザインの尊重、初回動線の検証、失敗から学んだ姿勢、など。
