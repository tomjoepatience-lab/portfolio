マネコ家計簿を作った話

2026年7月Web / TypeScript / three.js
🐱

「家計簿は続かない」。正直、僕自身がずっとそうでした。

きっかけは、実は別のアプリです。旅行の思い出を写真と一緒に残せる家計簿 「タビカケ」を作っていたとき、“写真で思い出が残ると、家計簿をつけるのが意外と楽しい” ことに気づきました。 だから最初に作っていたのは、それに近い「旅行用の家計簿」です。旅行中に写真を添えて記録し、あとでアルバムや振り返りレポートを作れる——そんなアプリでした。

でも、作りながら引っかかりました。旅行のときだけ使うアプリは、そもそも市場が小さい。 そして、普段の生活で使えない家計簿は、結局使われない。——ここで一度、大きく方向転換します。

「毎日使えるものにするには？」と考えたとき、日常の買い物の不便さに目が向きました。スーパーで合計金額だけ入力しても、あとで「これ、何を買ったんだっけ？」。 外食でいろいろ頼んでも、どの料理がいくらだったかは残らない。 だったら——レシートをそのまま取り込んで、勝手にジャンル分けして、支出を管理してくれるアプリがあればいい。それがマネコ家計簿の本当の出発点でした。

1. テーマ — レシートから始まって、“世帯まるごと”へ

出発点は、レシートを取り込んで自動でジャンル分けするという、 日常のための地味だけど効く機能でした。「金額だけ」では残らない“何を買ったか”を、レシートごと記録に変える。ここが土台です。

主役の猫「マネコ」は、名前のとおり招き猫から来ています。 こどもモードでは、代表キャラクターがいたほうが記録をつけるのが楽しくなると思って、前面に出しました。 一方おとなモードは、もう少し落ち着いた雰囲気で、レシート取込を中心に日常使いできるようにしています。

そして一番やりたかったのが親子連携です。親から子へお小遣いをあげられて、 子どもは小さいころから「お金の管理」を楽しく身につけられる。大人は子どものお金の使い方を見守りつつ、 自分の家計もまとめて管理できる。世帯の家計簿を一つで一括管理できるアプリにしたい、というのが芯にあるテーマです。

一番大事にしたのは、「子どもには楽しく、大人には便利に」。この2つを1つのアプリで両立させることでした。

2. サービスの説明 — 何ができるアプリか

初回ログインで、2つのモードを選びます。同じ家計データを別の見た目で使えます。

  • 🏘️ こどもモード「マネコタウン」 — 3Dの街で、おさいふ・コイン・XP/レベル・ごきげん度・きょうのチャレンジ・ちょきん箱・衣装ガチャ。家計簿をゲームのように続けられる。
  • 💼 おとなモード「マネコ家計簿」 — 「今月あと使える額」を大きく見せるヒーローカード＋予算バー＋つかいみち内訳＋最近の記録。覗きこむマネコがアクセント。

共通の下部ナビは ホーム / レポート / きろく / ちょきん / せってい。 支出は金額だけでなく、レシートOCR（Claudeの画像認識）で明細ごとに取り込めるので、「何を買ったか」まで残ります。 地図上のピンで「どこで使ったか」も記録でき、家族グループでの割り勘・精算にも対応。 親子でつなぐと、お小遣いのやり取りや子どものお金の管理も同じアプリの中で完結します。

3. 設計の説明

全体構成（3層）

授業の指定に沿って、フロント＝素のTypeScript（フレームワークなしのDOM操作）、 バック＝Node.js + Express、DB＝PostgreSQL の3層で作りました。 GitHubにpushするとRenderに自動デプロイされます。

  • フロント：src/client/。402×840のスマホ画面をキャンバスに見立て、素のDOMで描画。
  • サーバ：index.ts / api.ts / auth.ts / ocr.ts。cookieセッション認証、家族（親子）グループ、レシートOCR。
  • 3Dのマネコ：character.ts で、用意した招き猫のデザインを three.js を使ってホーム画面に配置。（キャラのデザイン自体はAIに作ってもらったものを取り込みました）

要になった設計判断：レシートOCR

このアプリの肝はレシート取込です。最初はAIを使わず、ブラウザ内で動くOCRライブラリ（Tesseract.js）で試しました。が、これが全然うまくいきませんでした。 レシートの細かい文字やレイアウトを正しく読めず、とても実用になりません。

そこでClaudeの画像認識（Haikuモデル）に切り替えたところ、レシートの情報を基本的にほぼすべて取り込めるようになりました。 レシートを宙に浮かせて撮ると少しズレが出ることはありますが、精度はかなり高く、「これでいこう」と決めました。 OCRエンジンを自前で頑張るより、画像認識AIに任せたほうが早くて正確、という判断です。

そのほかの設計

  • こども／おとなの2モードを1コードで：同じ家計データを、ログインユーザーの権限設定でこども向け・おとな向けに出し分けています。
  • PWA対応：manifest ＋ Service Worker で app shell をオフラインキャッシュし、ホーム画面から standalone 起動できる。?v=N でアセットをバージョニング。
  • スクロール設計：ホームは絶対配置＋scale、他タブは高さをフィットさせ内側だけスクロール、という切り分け。

4. 自分なりにこだわった点

① レシートを“実用”のレベルまで持っていったこと

ライブラリのOCRで諦めずにAIへ切り替え、「レシートを撮るだけで、何を買ったかまで記録に残る」体験を、実際に使えるレベルにできたこと。 手で品目を打ち込むより速いので、これなら続けられる、という手応えがありました。

② 子どもが“続けたくなる”仕掛け

家計簿はとにかく続かない。子どもに続けてもらうには、まず楽しくないといけない。楽しくするには、何か変化があったほうが食いつく—— そう考えて、貯金の進捗に合わせて景色が変わる「たびマップ」を作りました。 貯めるほど 商店街 → 都会 → タワマン → 飛行機 → 宇宙 と世界が広がり、ステージごとにマネコの衣装も変わります。 「記録＝作業」を「記録＝冒険が進む」に変えるのが狙いです。

③ 親子で“世帯まるごと”を一つに

子どもは楽しくお金の管理を覚えられて、大人は子どものお金遣いも家計もまとめて見られる。 お小遣いのやり取りまで同じアプリで完結させて、「家族のお金」を1つで扱えるようにしたことです。

5. 難しかった・悩んだこと

正直、最初からうまくいったわけではありません。むしろ失敗の連続でした。

デザインを“再現できない”

マネコのデザインは用意されていたのに、最初はそれを自己流に“解釈”して作ってしまい、「全然違う。忠実に再現して」と差し戻された。

デザインがあるのに再現できないのか、と最初はがっかりしました。 でも、使うAIモデルを変え、指示の出し方も「デザインに忠実に再現して」と変えて投げ直したら、今度はきちんと再現できた。あのときは素直に嬉しかったです。 正解が手元にあるときは、我流で解釈せず、まず忠実に再現する——という当たり前を学びました。

「AIがOK」でも、使うとバグだらけ

整ったデモアカウントでは動くのに、新規ユーザーが最初に触る画面はバグだらけだった （エラーでホームごと落ちてマネコが出ない、記録までの手順が多すぎる、など）。

これは実際に自分で踏みました。AIが「確認してOK」と言っても、実際に使ってみるとバグが出る——これは他のアプリを作っているときから感じていたことです。 なので慌てず、原因を落ち着いて切り分けて、フィードバックして直す。一度の会話ですべて片付くことは稀なので、それを何度も繰り返しながら、少しずつ理想の形に近づけていきました。

6. 学びになったと感じたこと

技術面で新しかったのは、デプロイ先の Render と、データベースの Neon（PostgreSQL） を初めて使ったことです。 デプロイやDBまわりの手数が増えたのは、素直に良い経験でした。 それ以外の実装は、これまでのアプリ開発で積み上げてきた知識でだいたい対応できた——新しく学ぶというより、持っているものを使った感覚でした。

むしろ大きかったのは作り方・進め方の学びです。ひとつは、 使うAIモデルや指示の出し方で、出てくるものの質がここまで変わるのかという驚き。開発の途中でモデルを変えたら、完成度が段違いになりました。 もうひとつは、AIが「OK」と言っても、自分で実際に使うまでは完成じゃないということ。この2つは、今後どんなものを作るときにも効いてくる学びだと思います。

7. 感想

振り返ると、最初からいいものが作れたわけではないし、コンセプトも最初は全然違いました（旅行用の家計簿でした）。 それを何度も考え直しながら、少しずつ自分のイメージに合う形へ近づけていく——作るというより、「育てる」に近い感覚でした。 うまくいかない前提で、方向転換もバグ潰しも繰り返す。その過程そのものが、いちばん面白かったところかもしれません。

これからは、決済連携で明細を自動で取り込めるようにしたり、月次レポートをA4などで出力できるようにしたりと、 もっと「日常でちゃんと使える家計簿」に育てていきたいです。
